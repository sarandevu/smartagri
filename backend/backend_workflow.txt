====================================================================
FARM DIGITAL TWIN - BACKEND WORKFLOW & ARCHITECTURE
====================================================================

This document explains the step-by-step working of the FastAPI backend
located in `app.py`.

--------------------------------------------------------------------
1. INITIALIZATION & SETUP
--------------------------------------------------------------------
- Framework: FastAPI is used to create a high-performance REST API.
- CORS Middleware: Allows the React frontend (running on a different 
  port) to communicate with the backend without security blocks.
- Database (SQLite): On startup, the `init_db()` function creates a 
  local file-based database (`farm.db`) with a single table:
  `sensor_logs`. This table stores all historical sensor readings, 
  predictions, and pump statuses.

--------------------------------------------------------------------
2. DATA VALIDATION (PYDANTIC MODELS)
--------------------------------------------------------------------
FastAPI uses Pydantic to ensure incoming requests have the correct format:
- `SensorPayload`: Validates incoming IoT sensor data (zone_id, moisture, 
  temp, humidity, light, rain_probability, mode).
- `PumpPayload`: Validates manual pump toggle requests (zone_id, action).

--------------------------------------------------------------------
3. CORE IRRIGATION LOGIC (THE "DIGITAL BRAIN")
--------------------------------------------------------------------
When sensor data arrives, it passes through 5 distinct decision functions:

  A) `detect_status(moisture)`:
     - Moisture < 30% -> CRITICAL
     - Moisture < 60% -> MODERATE
     - Otherwise -> GOOD

  B) `predict_soil(temperature, humidity)`:
     Estimates how fast the soil will dry based on current weather.
     - Temp > 32C & Hum < 40% -> FAST drying (Dry in 2 hours)
     - Temp > 30C -> MEDIUM drying (Dry in 4 hours)
     - Otherwise -> LOW drying (Stable)

  C) `weather_decision(rain_probability, moisture)`:
     Makes a smart irrigation choice.
     - If Rain > 70% -> DELAY (Save water, let it rain)
     - If Moisture < 30% -> IRRIGATE
     - Otherwise -> NO_ACTION

  D) `auto_control(mode, decision)`:
     - If the system is in "AUTO" mode and the decision is "IRRIGATE",
       it turns the pump "ON". Otherwise, "OFF".
       
  E) `flow_rate(pump_status)`:
     - Simulates water flow rate. If the pump is ON, it generates 
       a flow rate (L/min) to track water usage.

--------------------------------------------------------------------
4. API ENDPOINTS (ROUTES)
--------------------------------------------------------------------

1. POST /api/sensor-data
   - The primary endpoint. Receives IoT sensor data for a specific zone.
   - Runs the data through the core logic outlined above.
   - Saves the raw data AND the AI's decision/prediction into SQLite.
   - Returns the updated status back to the caller.

2. GET /api/zones
   - Fetches the absolute latest row for *every* unique zone.
   - Used by the frontend to render the distinct "Zone Cards".

3. GET /api/zone/{zone_id}/history
   - Fetches the last 50 historical data points for a specific zone.
   - Used to draw the Recharts Moisture/Temperature graph over time.

4. POST /api/pump
   - Used for the Manual Override button on the dashboard.
   - Fetches the latest state of the zone, forces the pump status to 
     whatever the user clicked (ON/OFF), sets mode to "MANUAL", and 
     saves a new database record.

5. GET /api/stats
   - Calculates aggregate metrics for the Dashboard Overview.
   - Sums up total water used across ALL logs.
   - Counts active alerts and active pumps from the latest known zone states.

--------------------------------------------------------------------
5. DATA FLOW SUMMARY
--------------------------------------------------------------------
ESP32/Frontend -> POST /api/sensor-data 
   -> Validated by Pydantic 
   -> Core Logic analyzes environment 
   -> Decision made (Irrigate/Delay) 
   -> SQLite database updated 
   -> Response sent back -> Frontend updates Dashboard UI
